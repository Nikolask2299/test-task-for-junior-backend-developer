# Тестовое задание на позицию junior backend разработчика

## Решение

Была реализована фича периодичности задач в модуле трекера.

### Предположения

- Периодические задачи создаются автоматически по расписанию (раз в день)
- Новая задача создаётся только если прошёл интервал с момента последнего создания
- Поля `start_date` и `end_date` опциональные, но если заданы, то должны быть логичны

### Ограничения

- Время проверки — UTC
- Задачи создаются с `status = new`

## Требования

- Go `1.23+`
- Docker и Docker Compose

## Быстрый запуск через Docker Compose

```bash
docker compose up --build
```

После запуска сервис будет доступен по адресу `http://localhost:8080`.

Если `postgres` уже запускался ранее со старой схемой, пересоздай volume:

```bash
docker compose down -v
docker compose up --build
```

Причина в том, что SQL-файл из `migrations/0001_create_tasks.up.sql` монтируется в `docker-entrypoint-initdb.d` и применяется только при инициализации пустого data volume.

### API

Базовый префикс API:

```text
/api/v1
```

#### Операции с периодическими задачами

- `POST /api/v1/tasks`
- `GET /api/v1/tasks`
- `GET /api/v1/tasks/{id}`
- `PUT /api/v1/tasks/{id}`
- `DELETE /api/v1/tasks/{id}`

- `POST /tasks/{id}/recurrences` — привязать существующую задачу к настройкам периодичности
- `POST /recurrences` — создать новую периодическую задачу с настройками
- `GET /recurrences/{id}` — получить настройки периодичности по ID
- `PUT /recurrences/{id}` — обновить настройки периодичности
- `DELETE /recurrences/{id}` — удалить настройки периодичности
- `GET /recurrences` — получить список всех настроек периодичности

#### Типы периодичности

- `daily` — ежедневная повторяемость
  - Параметр: `interval_days` — интервал в днях
- `monthly` — ежемесячная повторяемость
  - Параметр: `interval_months` — интервал в месяцах (1–30)
- `specific` — повторение в конкретные дни
  - Параметр: `specific_day` — массив дат
- `even_odd` — повторение по чётным/нечётным дням
  - Параметр: `even_odd_days` — `true` для чётных, `false` для нечётных

#### Общие параметры

- `start_date` — дата начала действия (опционально)
- `end_date` — дата окончания действия (опционально)

### Архитектура системы

Система построена по принципам чистой архитектуры (Clean Architecture), с чётким разделением на слои:

- **Transport** — HTTP-обработчики и маршрутизация (`internal/transport/http`)
- **UseCase** — бизнес-логика (`internal/usecase`)
- **Repository** — работа с базой данных (`internal/repository`)
- **Domain** — бизнес-сущности и правила (`internal/domain`)
- **Infrastructure** — внешние зависимости (база, планировщик)

#### Поток данных

1. HTTP-запрос поступает в обработчик (`handler`)
2. Обработчик валидирует входные данные и передаёт их в usecase
3. UseCase применяет бизнес-логику и взаимодействует с репозиторием
4. Репозиторий выполняет операции с базой данных
5. Результат возвращается через те же слои обратно клиенту

#### Планировщик

Ежедневно запускается задача, которая проверяет все активные периодические настройки и при необходимости создает новые экземпляры задач. Логика реализована в `internal/infrastructure/sheduler`.

#### Хранение данных

- `tasks` — основная таблица задач
- `recurrences` — настройки периодичности
- `recurrence_on_task` — связь между задачей и настройкой
